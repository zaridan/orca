# Non-terminal performance investigation: sidebar rows and git refresh

Generated by `pnpm exec vitest run --config config/vitest.non-terminal-perf.config.ts`.

## Scope

This investigation covers items 2 and 3 from `NON_TERMINAL_PERFORMANCE_BENCHMARK_IDEAS.md` and intentionally avoids Terminal/PTY implementation files. These are local opt-in benchmark samples, not CI pass/fail budgets.

## Item 2: sidebar/worktree-list row projection

Measured the pure row-building path used by `WorktreeList`: `buildRows(...)`, plus host-section projection where relevant. This does not claim full React commit time, but it isolates the deterministic CPU work that feeds the virtualized sidebar rows.

| Scenario | Worktrees | Output rows | Timing |
| --- | ---: | ---: | ---: |
| repo grouping, 10 repos × 100 worktrees | 1000 | 1061 | 0.45 ms median / 2.03 ms p95 / 0.66 ms mean (100 iterations) |
| status grouping, 10 repos × 100 worktrees | 1000 | 1053 | 0.51 ms median / 3.19 ms p95 / 0.76 ms mean (100 iterations) |
| PR grouping, 10 repos × 100 worktrees | 1000 | 1052 | 2.72 ms median / 4.82 ms p95 / 2.99 ms mean (100 iterations) |
| host sections, 10 repos × 100 worktrees × 3 hosts | 1000 | 1066 | 1.08 ms median / 4.2 ms p95 / 1.46 ms mean (100 iterations) |

### Item 2 findings

- The row projection does not look like a sustained bottleneck at 1,000 worktrees: medians were single-digit milliseconds in this sample, and the heavier grouping paths are the ones to watch in a full React/Electron profile.
- Active worktree switches should not rebuild `rows` because the main `useMemo(buildRows)` dependency list does not include `activeWorktreeId`; the active-card visual update is a separate path.
- The heavier measured path is PR grouping; host sections add another partitioning step but were not the slowest path in this sample.
- If users report sidebar lag at very large workspace counts, the next measurement should be a real React Profiler/Electron run for commit time and visible-row render counts. The pure projection is not obviously the bottleneck at 1,000 rows.

## Item 3: git/worktree refresh subprocess fanout

Measured actual `listWorktrees(repoPath)` calls against temporary git repositories with real linked worktrees. Git subprocess count is captured with a temporary PATH shim around `git`, while wall time is measured around the `listWorktrees` call itself.

| Scenario | Repos | Extra worktrees per repo | Returned worktrees | Git processes / iteration | Timing |
| --- | ---: | ---: | ---: | ---: | ---: |
| single repo refresh | 1 | 10 | 11 | 1 | 7.6 ms median / 8.17 ms p95 / 7.6 ms mean (20 iterations) |
| single repo refresh | 1 | 30 | 31 | 1 | 11.65 ms median / 14.12 ms p95 / 12.03 ms mean (20 iterations) |
| single repo refresh | 1 | 60 | 61 | 1 | 16.32 ms median / 19.02 ms p95 / 16.79 ms mean (20 iterations) |
| concurrent all-repo refresh | 5 | 10 | 55 | 5 | 16.05 ms median / 23.4 ms p95 / 18.48 ms mean (20 iterations) |

### Item 3 findings

- Current `listWorktrees` uses one `git worktree list --porcelain -z` subprocess per repo refresh on current git.
- Sparse-checkout annotation no longer adds N git subprocesses per worktree; it uses filesystem reads/stat through `resolveGitDir` + `info/sparse-checkout` checks.
- The remaining churn risk is repo fanout: `fetchAllWorktrees` and IPC event handlers can still refresh every repo concurrently, so total git process count scales with repo count rather than worktree count.
- If users report idle CPU from git refresh, benchmark the event source next: count how often `worktrees:changed`, startup hydration, and explicit refresh paths call all repos versus one touched repo.

## Suggested next actions

1. Add an Electron/React Profiler pass for item 2 only if we can reproduce sidebar interaction lag; the pure row projection does not currently look expensive enough to justify speculative optimization.
2. For item 3, instrument refresh callers to count repo-wide fanout over a 60-second idle/event window. The suspected issue is repeated refresh frequency, not per-worktree subprocess explosion.
3. Keep any future fixes outside Terminal/PTY files until the terminal model PRs settle.
